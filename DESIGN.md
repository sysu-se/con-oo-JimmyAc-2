# DESIGN.md

## 说明

这份文档按当前 `src/domain/index.js` 和已经接入后的 store / UI 代码来写。当前阶段我采用的是“复用 HW1 领域对象版本，再做最小调整”的思路，而不是重新完全写一套新设计。现在我已经先用 store adapter 的方式把领域对象接入到了现有 Svelte 流程里。

另外，这一版仍然保留了 HW1 里使用的工厂函数风格。这样做的原因是我希望保持原有设计和实现风格的连续性，避免在“接入 Svelte”之前先把领域对象整体重写一遍。当前代码层面最明显的调整，主要是 `Sudoku.toString()` 的外表化写法。我这里直接复用了项目原有 `src/node_modules/@sudoku/sudoku.js` 中 `printSudoku(...)` 的棋盘输出风格，只是改造成了返回字符串的 `toString()`。

## 一、`Sudoku` / `Game` 的职责边界

### 1. `Sudoku`

`Sudoku` 负责的是“当前盘面本身”。它内部持有一个 9x9 的 `grid`，并提供和盘面直接相关的操作：

- `getGrid()` / `grid`：返回当前盘面的深拷贝
- `canGuess(move)`：判断一个输入是否合法
- `guess(move)`：在合法时修改盘面
- `clone()`：复制出一个独立的盘面对象
- `toJSON()`：导出盘面的序列化结果
- `toString()`：导出可读字符串

我这里让 `Sudoku` 负责一部分数独规则校验，例如：

- 输入坐标范围是否合法
- 输入值是否在 0 到 9 之间
- 行、列、宫内是否冲突

也就是说，`Sudoku` 不再只是一个二维数组包装器，而开始自己保护一部分领域规则。

`Sudoku` 不负责的内容有：

- undo / redo 历史
- 一局游戏的会话状态
- Svelte 响应式更新

### 2. `Game`

`Game` 负责的是“一局游戏会话”。它持有当前 `Sudoku`，并在这个基础上管理历史：

- `getSudoku()` / `sudoku`：返回当前盘面的安全副本
- `guess(move)`：通过游戏对象执行一次正式输入
- `undo()` / `redo()`：撤销和重做
- `canUndo()` / `canRedo()`：判断历史栈是否可用
- `toJSON()`：把当前盘面和历史一起导出

所以这里的职责边界是：

- `Sudoku` 负责盘面和规则
- `Game` 负责会话和历史

这种分法和 HW1 基本保持一致，也就是继续沿用“`Sudoku` 管盘面，`Game` 管历史”的边界。区别在于，这一版已经进一步通过 store adapter 接入了真实界面流程。

## 二、`Move` 是值对象还是实体对象？为什么？

我认为 `Move` 是值对象，不是实体对象。

在我的设计里，`Move` 只是一次输入动作，例如：

```js
{ row: 0, col: 2, value: 4 }
```

它的意义完全由这三个字段决定。只要 `row`、`col`、`value` 一样，我就认为这是同一个 move。它不需要单独的 ID，也不需要自己有生命周期。

所以 `Move` 更像一个简单的数据载体，而不应该成为像 `Sudoku`、`Game` 那样的核心对象。

## 三、history 中存储的是什么？为什么？

我的 history 存的是 `Sudoku` 快照，也就是克隆出来的 `Sudoku` 对象。

具体来说：

- `undoStack` 里保存输入之前的盘面快照
- `redoStack` 里保存撤销时移出的当前盘面快照

这里我没有选择“存 move”，而是选择“存 snapshot”，原因是：

1. 恢复起来更直接。`undo()` / `redo()` 时直接恢复一个旧盘面就可以，不需要反推一步操作怎么撤销。
2. 对这次作业来说更稳。数独盘面只有 9x9，快照的空间开销可以接受，但能明显减少深浅拷贝带来的错误。

相比 HW1，这一版延续了原来的 snapshot history 思路，仍然由 `Game` 负责管理 `undoStack` / `redoStack`。这样做的重点不是重新发明 history 结构，而是保证它能够继续用于真实接入。

## 四、我的复制策略是什么？哪些地方需要深拷贝？

我的复制策略是：只要涉及盘面数据的接收、返回、复制、历史保存和导出，就都使用深拷贝。

我现在用的深拷贝方式比较直接：

```js
grid.map(row => [...row])
```

因为 `grid` 是二维数组，如果只做浅拷贝，内层数组还是共用的，后面修改时就会互相影响。

### 1. `createSudoku(input)`

- 是否必须深拷贝：必须。
- 原因：`input` 是外部传进来的盘面，不能直接和对象内部状态共用同一份二维数组。
- 如果误用了浅拷贝：外部后续修改 `input` 时，`Sudoku` 内部盘面也会一起变化，对象封装会失效。

### 2. `getGrid()` / `grid`

- 是否必须深拷贝：必须。
- 原因：返回给外部的是盘面快照，而不是内部真实状态。
- 如果误用了浅拷贝：外部就可以绕过 `guess()` 直接修改对象内部盘面。

### 3. `clone()`

- 是否必须深拷贝：必须。
- 原因：clone 的目标就是产生完全独立的新对象。
- 如果误用了浅拷贝：原对象和 clone 会共享内层数组，修改一个会污染另一个。

### 4. `createSudokuFromJSON(json)`

- 是否必须深拷贝：必须。
- 原因：反序列化时也要重新构造独立内部状态，而不是继续和外部 JSON 数据共享。
- 如果误用了浅拷贝：恢复出来的 `Sudoku` 可能会被外部传入的 JSON 数据继续影响。

### 5. `createGame({ sudoku })`

- 是否必须深拷贝：必须。
- 原因：`Game` 内部持有的是一局游戏自己的当前盘面，不应该继续和外部传入的 `Sudoku` 共用同一个实例。
- 如果误用了浅拷贝：外部如果还持有原来的 `Sudoku` 引用并继续修改，会破坏 `Game` 内部状态和历史的一致性。

### 6. `getSudoku()` / `sudoku`

- 是否必须深拷贝：我这里选择必须。
- 原因：这样外部拿到的只是当前盘面的副本，而不是 `Game` 内部真正持有的那一个对象。
- 如果误用了浅拷贝：外部可以直接调用 `game.getSudoku().guess(...)` 改盘面，但这一步不会进入 `Game.guess()`，history 也不会同步记录，Undo / Redo 就可能失效。

### 7. `Game.guess(move)`

- 是否必须深拷贝：必须。
- 原因：这里要把当前 `Sudoku` 快照压入 `undoStack`。
- 如果误用了浅拷贝：后续对当前盘面的修改会把 `undoStack` 中的历史也一起改掉，撤销时恢复不到真正的旧状态。

### 8. `undo()`

- 是否必须深拷贝：必须。
- 原因：当前状态要保存到 `redoStack`，以后还要拿出来做 `redo()`。
- 如果误用了浅拷贝：`redoStack` 里的历史会随着当前盘面继续变化而被污染。

### 9. `redo()`

- 是否必须深拷贝：必须。
- 原因：当前状态要重新压回 `undoStack`，供后续继续撤销。
- 如果误用了浅拷贝：连续多次 undo / redo 后，历史栈里的状态会越来越不可靠。

### 10. `toJSON()`

- 是否必须深拷贝：必须。
- 原因：导出的是一份独立的可保存数据，而不是对象内部真实状态。
- 如果误用了浅拷贝：外部拿到序列化结果继续修改，也可能反过来污染当前对象状态。

总结来说，这次作业里最容易出 bug 的地方就是二维数组共享引用问题。只要某一处误用了浅拷贝，就可能导致：

- 输入污染内部状态
- clone 不独立
- history 被污染
- undo / redo 恢复错误状态

## 五、我的序列化 / 反序列化设计是什么？

### 1. `Sudoku`

`Sudoku.toJSON()` 返回：

```js
{ grid: number[][] }
```

也就是说，`Sudoku` 的序列化结果是一个普通对象，里面只保存当前盘面的二维数组。

`createSudokuFromJSON(json)` 会从这个对象中取出 `grid`，然后重新构造一个新的 `Sudoku`。

为了兼容 HW1 的旧测试接口，我这里也允许直接传入二维数组。也就是说现在支持两种恢复方式：

- `createSudokuFromJSON({ grid })`
- `createSudokuFromJSON(grid)`

### 2. `Game`

`Game.toJSON()` 返回：

```js
{
  sudoku: { grid: ... },
  history: {
    undo: [...],
    redo: [...]
  }
}
```

这里的设计重点仍然是让 `Game` 的 JSON 保留自己的语义，也就是明确表达“这是一个 game，它里面持有一个 sudoku，并且有对应 history”。这样后面即使做真实接入，序列化结构也不会和单独的 `Sudoku` 完全混在一起。

`createGameFromJSON(json)` 会：

1. 先恢复当前 `Sudoku`
2. 再恢复 `undoStack`
3. 再恢复 `redoStack`
4. 最后创建新的 `Game`

同样，为了兼容 HW1 旧接口，我这里也接受旧格式：

- 如果传进来的是一个二维数组，或者一个只有 `grid` 的对象，就把它当成“只保存当前盘面”的旧版 `Game JSON`

### 3. 哪些字段会被序列化，哪些不会？

会被序列化的：

- `Sudoku` 当前盘面 `grid`
- `Game` 当前 `sudoku`
- `Game` 的 `undoStack`
- `Game` 的 `redoStack`

不会被序列化的：

- 各个对象的方法本身
- 闭包内部的实现细节
- 校验函数、辅助函数这些运行期逻辑

## 六、我的外表化接口是什么？为什么这样设计？

我这里主要提供了两个外表化接口：`toJSON()` 和 `toString()`。

### 1. `toJSON()`

`toJSON()` 主要给程序使用，用于：

- `JSON.stringify(...)`
- 保存当前状态
- 反序列化恢复

这里的特点是：

- `Sudoku` 只导出盘面
- `Game` 导出盘面加 history

这样区分后，两个对象在序列化层面的边界更清楚。

### 2. `toString()`

`toString()` 主要给人看，用于调试。

这里我没有继续沿用 HW1 里那种简单的按行拼接二维数组字符串的方式，而是直接复用了项目原有 `src/node_modules/@sudoku/sudoku.js` 中 `printSudoku(...)` 的棋盘打印风格.

这样设计的原因是：

- 比默认的 `[object Object]` 清楚得多
- 调试时能直接看出当前局面
- 复用了原项目本来就有的棋盘输出风格, 和原工程整体更一致
- 加上了 3x3 宫分隔和边框, 可读性比只输出二维数组更好

## 七、相比 HW1，我做了什么调整？

相比 HW1，我这次没有选择整体推翻原来的设计，而是以复用为主，只做了和当前阶段目标最相关的调整。

1. 保留了原本的工厂函数风格，继续使用 `createSudoku(...)` 和 `createGame(...)`。
2. 保留了原本的职责边界，也就是 `Sudoku` 管盘面，`Game` 管 history。
3. 保留了 snapshot 的 Undo / Redo 实现方式。
4. 对外表化部分做了调整，尤其是 `Sudoku.toString()`。这里不是重新设计一种全新的文本格式，而是复用了原项目 `printSudoku(...)` 的输出风格，让盘面输出更适合调试和阅读。

所以这一步的重点不是“重写领域对象”，而是“先保持原有设计稳定，再完成最小正确接入”。

## 八、为什么 HW1 的做法不足以支撑真实接入？

HW1 的版本虽然已经通过了测试，但它主要还是“测试可用的领域对象”。这次 `con-oo` 的要求更进一步，重点已经不是单独通过测试，而是要让领域对象真正进入真实使用流程。

所以 HW1 做法目前最大的不足不是“完全不能用”，而是：

1. 领域对象虽然存在，但还没有真正成为 view 层消费的核心。
2. UI 和领域对象之间还缺少清晰的桥接层。
3. 仅靠 HW1 版本本身，还不足以说明 Svelte 响应式更新是如何发生的。

也就是说，这次作业的重点已经从“设计出领域对象”转向“让领域对象真正被消费”。

## 九、我的新设计有哪些 trade-off？

这次设计也有 trade-off。

1. 我选择继续复用 HW1 的对象设计，优点是连续性好，缺点是这一步改动不会特别大。
2. 我选择继续存 snapshot 而不是存 move，优点是简单稳妥，缺点是 history 会多占一些空间。
3. 我保留工厂函数风格而不是直接改成 class，优点是和 HW1 连续，缺点是不能字面上使用 `#private` 语法。
4. 我采用的是最小接入方案，也就是优先让输入、Undo / Redo 和当前盘面显示先走通。优点是改动范围小，缺点是 adapter 设计还可以继续整理得更漂亮。

## 十、领域对象如何被消费

这一步我采用了文档里推荐的 **Store Adapter** 方案。也就是说，View 层并不直接拿 `Game` 对象本身，而是消费一个面向 Svelte 的 store。

### 1. View 层直接消费的是什么？

View 层当前直接消费的是 `userGrid` 这个 custom store，而不是直接消费原始 `Game` 对象。

这个 store 的位置在：

- `src/node_modules/@sudoku/stores/grid.js`

它内部持有一个 `Game`：

- 当基础题面 `grid` 变化时，adapter 会重新创建一个新的 `Game`
- 平时输入、Undo、Redo 都通过这个内部 `Game` 进行

### 2. View 层拿到的数据是什么？

当前 View 层拿到的主要数据有：

- `$userGrid`：当前用户看到的 9x9 盘面
- `undoAvailable` / `redoAvailable`：按钮是否可用
- `$invalidCells`：由当前盘面导出的冲突格子

其中最核心的是 `$userGrid`。它不是旧逻辑自己维护的一份独立数组，而是每次从 `Game` 当前持有的 `Sudoku` 中同步出来的视图状态。

### 3. 用户操作如何进入领域对象？

当前链路是：

1. 用户在组件里点击按钮或按键
2. 组件调用 `userGrid` 暴露的方法
3. `userGrid` 内部再去调用 `Game` / `Sudoku`

具体来说：

- 键盘输入：
  - `Keyboard.svelte` 调用 `userGrid.guess($cursor, num)`
  - adapter 内部调用 `game.guess({ row, col, value })`

- Undo / Redo：
  - `Actions.svelte` 调用 `userGrid.undo()` / `userGrid.redo()`
  - adapter 内部调用 `game.undo()` / `game.redo()`

也就是说，现在用户操作已经不是直接改二维数组，而是先进入 adapter，再进入领域对象。

### 4. 领域对象变化后，Svelte 为什么会更新？

原因是 adapter 在每次领域对象变化之后，都会重新把当前 `Game` 中的盘面同步到 Svelte store：

- 通过 `userGrid.set(...)`
- 把 `game.getSudoku().getGrid()` 这份新数据写回去

Svelte 对 store 的 `set(...)` 是响应式的，所以 `$userGrid` 会刷新，界面也会跟着刷新。

## 十一、响应式机制说明

### 1. 我依赖的是什么机制？

我现在依赖的是：

- custom store
- `$store`
- 每次对象变化后重新 `set(...)` 一份新的响应式状态

也就是说，我没有让 Svelte 直接观察领域对象内部字段，而是通过 store adapter 显式地把最新状态重新发布给 View。

### 2. 哪些数据是响应式暴露给 UI 的？

当前响应式暴露给 UI 的主要有：

- `userGrid`
- `invalidCells`
- undo / redo 是否可用的状态

其中：

- `userGrid` 是最核心的盘面响应式状态
- `invalidCells` 是基于当前盘面推导出来的视图状态

### 3. 哪些状态留在领域对象内部？

留在领域对象内部的包括：

- `Sudoku` 内部真实盘面
- `Game` 内部当前 `Sudoku`
- `undoStack`
- `redoStack`

UI 不会直接改这些内部对象，而是通过 adapter 暴露的命令接口来驱动变化。

### 4. 如果不用这个方案，而是直接 mutate 内部对象，会出现什么问题？

如果直接 mutate 领域对象内部状态，而没有重新通过 store 发布新状态，那么 Svelte 不一定知道该刷新。因为当前这套响应式链路依赖的是：

- store 的 `set(...)`
- `$store` 订阅结果变化

如果只是对象内部字段悄悄变了，但外层 store 没有重新发布，界面就可能出现“数据已经变了，但 UI 没刷新”的问题。

所以这次我采用 adapter 的原因就是：

- 领域对象内部继续按对象方式管理状态
- View 层只消费 adapter 暴露出的响应式结果
